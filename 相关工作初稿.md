## 2. 相关工作

本研究采用两阶段技术路线：首先在漏洞代码数据集上训练安全感知编码器，然后将其应用于CoT推理路径的安全选择。本章从代码漏洞检测、思维链推理与路径选择、大语言模型安全、对比学习与代码表示四个方面综述相关工作，并阐述本文与现有研究的差异。

### 2.1 代码漏洞检测方法

代码漏洞检测是软件安全领域的核心问题。传统的静态分析方法通过规则匹配和数据流分析检测漏洞，代表性工具包括Bandit、Semgrep、CodeQL等。然而，这类方法存在规则覆盖不全、误报率高等问题。

近年来，深度学习方法在漏洞检测领域取得显著进展。Shiri Harzevili等人[1]系统综述了机器学习在软件漏洞检测中的应用，分析138篇论文后发现**图表示(57.2%)和深度学习模型(88.4%)**是主流方法。Li等人[2]提出VulDeePecker，首次将深度学习应用于漏洞检测，使用代码小片段作为输入。Zhou等人[3]提出Devign，结合图神经网络和抽象语法树(AST)进行漏洞检测。Chakraborty等人[4]构建ReVeal数据集，揭示了现有漏洞检测模型在实际场景中的局限性。

在代码表示方面，Feng等人[5]提出CodeBERT，是首个面向编程语言的预训练模型，在代码搜索、摘要等任务上取得显著效果。Guo等人[6]进一步提出GraphCodeBERT，引入数据流图信息，在代码生成和漏洞检测任务上表现更优。Fu等人[7]对GitHub Copilot生成的代码进行实证研究，发现**27.3%的代码片段包含安全漏洞**，涉及43种CWE类型。

Ding等人[8]构建PrimeVul高质量漏洞数据集，揭示了现有数据集标签准确率低、重复率高等问题，并指出SOTA模型在BigVul上可达68% F1，但在严格数据集上仅3% F1。这一发现表明**数据质量对漏洞检测模型的评估至关重要**。

上述研究主要关注漏洞检测任务本身，本文则将漏洞检测能力作为基础，进一步将其应用于**推理路径的安全评估**场景。

### 2.2 思维链推理与路径选择

思维链(Chain-of-Thought, CoT)推理是一种引导大语言模型生成中间推理步骤的技术，由Wei等人[9]首次提出，显著提升了LLM在复杂推理任务上的表现。Wang等人[10]提出的Self-Consistency方法通过多数投票选择最频繁答案，成为CoT路径选择的主流方法。然而，这种方法仅关注答案频率，无法区分路径质量。

近年来，研究者开始关注CoT路径选择的改进。Taubenfeld等人[11]提出CISC方法，通过置信度加权投票减少采样需求，证明了连续评分机制优于离散投票。Wang等人[12]提出的Soft-SC方法进一步验证了连续似然分数在多答案任务中的有效性。Tan等人[13]揭示了Self-Consistency的核心弱点——"自一致错误"问题，即模型多次采样都生成相同错误响应，并提出了跨模型探针方法进行检测。

在CoT推理的理论基础方面，Korbak等人[14]提出CoT可监控性概念，论证了通过监控模型推理过程检测不良意图的可能性，为本文的安全路径选择方法提供了重要的理论支撑。该研究同时指出这种可监控性是脆弱的，需与其他安全方法配合使用。Liang等人[15]系统综述了LLM的内部一致性问题，提出Self-Feedback框架，为一致性评估提供了理论基础。Min等人[16]提出的IdentityChain框架通过"代码→规范→代码"的循环一致性检查验证代码正确性，其循环验证思想可借鉴于安全路径的验证。

在代码生成场景中，路径选择问题尤为重要。Wang等人[17]构建CodeSecEval数据集，系统评估了LLM在代码生成任务中的安全性表现，发现当前模型经常忽视安全问题并生成易受攻击的代码。Ren等人[18]提出CodeAttack框架，揭示代码输入可绕过LLM安全防护，攻击成功率超过80%。

**本文的核心创新**：首次将安全感知引入**代码生成任务的CoT路径选择**。

**应用场景**：当用户向LLM提出代码生成请求时，LLM通过CoT推理生成多条候选代码，本文方法从中选择最安全的代码输出。

**技术路线**：
```
阶段一：安全感知编码器训练
├── 数据：漏洞代码(func_before) vs 修复代码(func_after)
├── 方法：对比学习，学习区分安全/危险代码
└── 输出：安全感知编码器

阶段二：CoT路径安全选择
├── 输入：用户代码生成请求
├── LLM采样：生成K条推理路径，每条输出一段代码
├── 安全评估：编码器评估每段代码的安全性
└── 输出：选择安全得分最高的代码
```

**示例**：
```
用户请求："帮我写一个根据用户ID查询用户数据的函数"

路径A生成代码（危险）：
query = f"SELECT * FROM users WHERE id={user_id}"  # SQL注入漏洞

路径B生成代码（安全）：
cursor.execute("SELECT * FROM users WHERE id=?", (user_id,))  # 参数化查询

编码器评估：路径A距离漏洞原型近，路径B距离安全原型近
最终选择：路径B
```

这种方法将漏洞检测能力与代码生成的推理路径选择相结合，填补了现有研究的空白。

### 2.3 大语言模型安全

LLM安全研究涵盖对抗攻击、越狱、安全对齐等多个方向。Yao等人[19]系统综述了LLM安全与隐私领域的281篇论文，提出了Good/Bad/Ugly三分法框架。Sai等人[20]综述了生成式AI在网络安全领域的15+应用场景。

针对推理模型的安全性，Kuo等人[21]提出H-CoT攻击方法，劫持CoT安全推理机制，将拒绝率从98%降至2%以下，揭示了现有安全机制的脆弱性。Zhang等人[22]系统研究了大型推理模型(LRM)的安全增强，发现推理能力增强不等于安全性提升，甚至可能降低。Jiang等人[23]首次系统研究LRM安全性，发现长CoT推理并不保证安全输出。

在安全推理数据生成方面，Kumarage等人[24]提出AIDSAFE方法，通过多智能体审议生成安全推理数据，用于微调模型以增强安全泛化能力。该方法关注训练阶段的安全数据生成，与本文推理阶段的安全路径选择形成互补。Luo等人[25]提出AgentAuditor框架，实现了人类水平的LLM智能体安全评估，为安全评估方法提供了参考。

本文与上述研究形成互补：现有工作多关注**训练阶段**的安全对齐或**攻击防御**，本文聚焦**推理阶段**的安全路径选择，提供了一种轻量级、可解释的安全增强方法。

### 2.4 对比学习与代码表示

对比学习是一种有效的表示学习方法，通过学习样本间的相似性构建语义空间。Gao等人[26]提出的SimCSE方法证明了Dropout可作为最小数据增强，在句子嵌入任务上取得显著效果。Snell等人[27]提出的原型网络通过计算到类别原型表示的距离进行分类，在小样本学习场景表现优异。

在代码表示领域，Zhang等人[28]将LLM与对比学习结合，提出LA-UCL框架用于少样本文本分类。Xu等人[29]将对比学习应用于代码领域，提出CodeGPTSensor检测器区分LLM生成代码与人类编写代码。Dai等人[30]对比了对比学习(CL)与监督微调(SFT)在LLM重排序任务中的表现。

本文借鉴SimCSE的对比学习框架和原型网络的原型构建思想，创新性地将其应用于**安全感知的代码表示学习**。具体而言：
- **正样本对**：同一漏洞的不同代码实例（学习漏洞模式的通用表示）
- **负样本对**：漏洞代码 vs 修复后代码（学习安全/危险代码的边界）
- **原型构建**：为每种CWE类型构建漏洞原型和安全原型

通过这种方式，编码器学习区分安全代码与漏洞代码的表示，为后续的CoT路径安全评估提供基础。

### 2.5 本文定位

综上所述，现有研究存在以下不足：

| 方向 | 现有研究 | 不足之处 | 本文创新 |
|------|----------|----------|----------|
| 代码漏洞检测 | VulDeePecker、Devign、静态分析 | 仅关注检测，未应用于推理场景 | 应用于推理场景 |
| CoT路径选择 | Self-Consistency、CISC、Soft-SC | 未考虑安全性维度 | 安全性维度 |
| LLM安全 | 安全对齐、越狱防御 | 多关注训练阶段，推理阶段研究不足 | 推理阶段轻量级防御 |

本文提出**SecureProtoCoT方法**，采用两阶段技术路线：
1. **阶段一**：在漏洞代码数据集（MSR_data_cleaned，188,636函数，6种CWE类型）上训练安全感知编码器
2. **阶段二**：将编码器集成到CoT推理框架，实现安全感知的路径选择

这一方法将漏洞检测能力与推理路径选择相结合，填补了上述研究空白，为LLM代码生成的安全性提供了一种轻量级、可解释的解决方案。
