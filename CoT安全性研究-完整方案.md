### 0.2 核心叙事框架

#### 第一步：发现问题（你的独特贡献）

> 在AI编程助手的代码生成任务中，我们发现一个系统性现象：LLM在生成代码时，可能输出多条不同的推理路径，其中某些路径生成的代码虽然功能正确，但存在安全漏洞（如SQL注入、缓冲区溢出等）。现有CoT路径选择方法（如Self-Consistency）无法识别这种安全风险。

#### 第二步：定义问题

> 现有CoT路径选择方法（如Self-Consistency）只关注答案频率或语义相似度，无法识别代码安全性。在代码生成场景中，我们需要一种新的机制：从多条候选代码中选择最安全的版本。

#### 第三步：解决问题

> 本文提出SecurePath框架，采用两阶段技术路线：
> 1. **阶段一**：在漏洞代码数据集上训练安全感知编码器，学习区分漏洞代码与安全代码
> 2. **阶段二**：将编码器集成到CoT推理框架中，从多条候选代码中选择安全得分最高的代码

#### 第四步：验证效果

> 采用双实验设计互为印证：
> - 实验A（大规模自动评估）：在MSR测试集上验证原型的代码安全性判别力
> - 实验B（端到端验证）：在LLM实际代码生成场景中验证方法可用性

### 0.2 论文叙事框架（5章）

#### Chapter 1: 发现问题 — "LLM 生成的代码可能功能正确但存在漏洞"

**切入**：从实习中的真实案例出发——同一个"查询用户数据"需求，Sonnet 生成了 f-string SQL 拼接（可 SQL 注入），Opus 生成了参数化查询（安全）。这不是随机 bug，而是模型的**系统性差异**。

**引出现有方法的盲区**：Self-Consistency（多数投票）只看答案频率，不考虑安全性。一段代码被 8/10 条路径生成，不代表它安全——可能 8 条都忽视了同一个漏洞。

**核心故事钩子**：CoT 路径选择需要"安全意识"，现有方法全都没有。

#### Chapter 2: 相关工作 — "漏洞检测和路径选择是两条平行线"

| 领域 | 做什么 | 局限 |
|------|--------|------|
| 代码漏洞检测 | 静态分析、深度学习、CodeBERT fine-tune | 只管"有没有漏洞"，不参与路径选择 |
| CoT 路径选择 | Self-Consistency、CISC、Soft-SC | 只管"哪个答案最可能对"，不考虑安全 |

**Gap**：没有人把漏洞检测能力引入路径选择。本文填补这个空白。

#### Chapter 3: 方法 — "两阶段：先学会看代码安不安全，再用来选路径"

**阶段一：安全感知编码器训练**
- 在 MSR 数据集上使用 CodeBERT + 对比学习 + InfoNCE 损失
- 关键挑战：vul=0 记录中 func_before == func_after（92.7%），导致训练崩溃（准确率仅 52%）
- 解决方法（方案二）：从不同函数构建漏洞/安全代码池，确保语义差异
- 结果：准确率从 52% → 85.7%，构建 vul_prototype 和 safe_prototype

**阶段二：原型驱动的安全路径选择**
- 用户提出代码生成请求 → LLM 生成 K 条候选代码
- 对每条候选计算 SafetyScore = cosine(emb, safe_proto) - cosine(emb, vul_proto)
- 选择得分最高的代码输出

#### Chapter 4: 实验 — "双实验设计，互为印证"

## 一、研究背景与动机

### 1.1 应用场景：代码生成任务的安全问题

**应用场景定义**：本文聚焦于LLM代码生成任务中的安全问题。

当用户向LLM提出代码生成请求时：
```
用户："帮我写一个根据用户ID查询用户数据的函数"

LLM可能生成多条候选代码：

路径A输出：
def get_user(user_id):
    query = f"SELECT * FROM users WHERE id={user_id}"  # SQL注入漏洞！
    return db.execute(query)

路径B输出：
def get_user(user_id):
    query = "SELECT * FROM users WHERE id=?"
    return db.execute(query, (user_id,))  # 安全：参数化查询
```

**核心问题**：现有CoT路径选择方法无法识别路径A存在SQL注入漏洞，可能选择危险代码。

### 1.2 MSR_data_cleaned数据集分析

数据集揭示的现象：
- 总函数数：188,636
- 漏洞函数：10,900 (5.8%)
- 非漏洞函数：177,736 (94.2%)

**6种内存安全漏洞分布**：
| CWE | 漏洞类型 | 数量 | 占比 |
|-----|----------|------|------|
| CWE-119 | 缓冲区溢出 | 26,497 | 48.7% |
| CWE-416 | Use After Free | 9,780 | 18.0% |
| CWE-125 | 越界读取 | 7,717 | 14.2% |
| CWE-476 | 空指针解引用 | 4,572 | 8.4% |
| CWE-190 | 整数溢出 | 3,671 | 6.7% |
| CWE-787 | 越界写入 | 2,918 | 5.4% |

### 1.3 研究机会

通过对比学习，可以让模型学习区分漏洞代码与安全代码的表示：

- **训练数据**：漏洞代码(func_before) vs 修复代码(func_after)
- **学习方法**：对比学习，让漏洞代码聚集，安全代码聚集，两者分离
- **应用**：评估LLM生成的多条候选代码，选择最安全的版本

**核心假设**：漏洞代码和安全代码在向量空间中具有不同的分布特征，通过对比学习可以有效区分。

### 3.2 阶段一：安全感知编码器训练

#### 3.2.1 数据准备

**数据来源**：MSR_data_cleaned（188,636函数）

**采样策略**：每种CWE类型采样1000条，共6000条样本

**标签构建**：
- 漏洞代码(func_before) → label=1 (危险)
- 修复代码(func_after) → label=0 (安全)

**数据划分**：
- 策略一（RQ3）：按比例划分（训练70%/验证10%/测试20%）
- 策略二（RQ4）：留一法（5种CWE训练，1种CWE测试）

#### 3.2.2 对比学习框架

**编码器**：CodeBERT-base

**投影头**：768维 → 256维

**对比损失**：InfoNCE Loss

```
正样本对：同类漏洞代码聚集
负样本对：漏洞代码 vs 修复代码分离
```

**训练目标**：
- 漏洞代码在向量空间中聚集（形成漏洞原型）
- 安全代码在向量空间中聚集（形成安全原型）
- 两者在向量空间中分离

#### 3.2.3 原型构建

**漏洞原型**：各CWE类型漏洞代码向量的均值
$$p_{vul}^{(cwe)} = \frac{1}{N}\sum_{i=1}^{N} Encoder(code_{vul}^{(i)})$$

**安全原型**：各CWE类型修复代码向量的均值
$$p_{safe}^{(cwe)} = \frac{1}{N}\sum_{i=1}^{N} Encoder(code_{safe}^{(i)})$$

### 3.3 阶段二：CoT路径安全选择

#### 3.3.1 应用流程

```
1. 用户提出代码生成请求

2. LLM采样生成K条推理路径
   路径1 → 代码1
   路径2 → 代码2
   ...
   路径K → 代码K

3. 编码器评估每段代码的安全性
   SafetyScore(code_i) = sim(Encoder(code_i), p_{safe})

4. 选择安全得分最高的代码输出
   code* = argmax SafetyScore(code_i)

### 3.2 第一阶段：安全感知表示学习

#### 3.2.1 多粒度表示提取

**输入**：
- 输入问题 $q$
- K条推理路径 $\{c_1, c_2, ..., c_K\}$

**处理**：
1. 调用冻结的LLM生成推理路径
2. 使用代码专用编码器（**CodeBERT/GraphCodeBERT**）编码
3. 步骤分割：按换行符或逻辑标记分割
4. 步骤级表示：$\{s_{i,1}, s_{i,2}, ..., s_{i,M}\}$
5. 路径级表示：$z_i = \text{MeanPool}(\{s_{i,m}\})$

**编码器选择依据**：
- CodeBERT/GraphCodeBERT在代码表示任务上优于通用BERT
- 图表示（CPG/AST）在漏洞检测中应用最广（57.2%研究使用）
- 可选扩展：使用Joern工具生成代码属性图(CPG)作为输入

#### 3.2.2 安全标签注入

**核心创新**：引入安全标签作为监督信号

**标签来源**：
1. **自动漏洞检测器**：
   - SQL注入检测（Bandit, Semgrep）
   - XSS检测（ESLint安全规则）
   - 敏感信息泄露检测
   
2. **专家标注**：
   - 对自动检测不确定的样本进行人工审核
   - 构建安全推理模式知识库

**标签定义**：
- 安全路径 $P_{safe}$：通过所有安全检测，无漏洞
- 危险路径 $P_{danger}$：存在至少一个安全漏洞
- 可疑路径 $P_{suspicious}$：存在潜在风险，需人工确认

#### 3.2.3 步骤级安全对比对齐

**损失函数设计**：

$$
\mathcal{L}_{safe} = -\log \frac{\exp(\text{sim}(z_q, s^+_{safe})/\tau)}{\sum_{s^- \in P_{danger}} \exp(\text{sim}(z_q, s^-)/\tau)}
$$

其中：
- $z_q$：问题表示（锚点）
- $s^+_{safe}$：安全路径的步骤表示（正样本）
- $s^-$：危险路径的步骤表示（负样本）
- $\tau$：温度参数

**对齐目标**：
- 安全推理步骤向问题表示靠拢
- 危险推理步骤远离问题表示
- 形成安全/危险路径的分离簇

### 3.3 第二阶段：安全原型构建

#### 3.3.1 安全路径加权聚合

$$
p_{safe} = \sum_{c_i \in P_{safe}} w_i \cdot z_i
$$

其中权重：
$$
w_i = \frac{\exp(\text{sim}(z_q, z_i))}{\sum_{j \in P_{safe}} \exp(\text{sim}(z_q, z_j))}
$$

**直觉**：安全原型是所有安全路径的加权平均，权重由与问题的相似度决定。

#### 3.3.2 危险路径特征提取

对于每条危险路径，提取其危险特征：

$$
f_{danger}^{(i)} = z_i - p_{safe}
$$

**含义**：危险特征 = 危险路径表示 - 安全原型（表示该路径偏离安全模式的程度）

#### 3.3.3 安全-危险原型对比

构建安全边界：

$$
\text{boundary}(c_i) = \|z_i - p_{safe}\|_2
$$

**阈值确定**：
- 在验证集上确定最优阈值 $\theta$
- 如果 $\text{boundary}(c_i) > \theta$，则标记为危险路径

### 3.4 第三阶段：安全路径选择

#### 3.4.1 路径安全得分计算

$$
\text{SafetyScore}(c_i) = \text{sim}(z_i, p_{safe}) + \lambda \cdot \text{Confidence}(c_i) - \alpha \cdot \text{RiskScore}(c_i)
$$

其中：
- $\text{sim}(z_i, p_{safe})$：与安全原型的相似度
- $\text{Confidence}(c_i)$：模型生成该路径的置信度（参考CISC）
- $\text{RiskScore}(c_i)$：漏洞检测器的风险评分
- $\lambda$：置信度权重系数
- $\alpha$：风险权重系数

**置信度引入依据**（参考CISC论文发现）：
- 置信度加权可减少40%以上的采样需求
- 连续评分机制比离散投票更灵活
- 置信度+安全性联合评分可提升路径选择质量

#### 3.4.2 安全阈值过滤

```
if SafetyScore(c_i) < threshold_safe:
    标记为危险，排除
elif SafetyScore(c_i) < threshold_suspicious:
    标记为可疑，需人工确认
else:
    标记为安全，保留
```

#### 3.4.3 最优安全路径选择

从保留的安全路径中选择：

$$
c^* = \arg\max_{c_i \in P_{safe}} \text{SafetyScore}(c_i)
$$

### 3.5 可选扩展

#### 3.5.1 图表示增强（参考Systematic Review发现）

**背景**：57.2%的漏洞检测研究使用图表示（CPG/AST）

**可选方法**：
- 使用Joern工具生成代码属性图(CPG)
- 结合控制流图(CFG)和数据流图(DFG)
- 用图神经网络(GNN)编码结构信息

**优点**：
- 捕获代码结构信息（比序列表示更丰富）
- 可检测跨语句的数据流漏洞

**缺点**：
- 计算开销增加
- 需要额外的代码解析工具

#### 3.5.2 多模型验证机制（参考Too Consistent to Detect）

**背景**：单一模型的自一致错误难以检测

**可选方法**：
- 引入外部验证模型（如小型安全检测器）
- 跨模型探针：融合多个模型的隐藏状态证据

**优点**：
- 提升自一致错误的检测能力
- 增强方法的鲁棒性

**缺点**：
- 增加系统复杂度
- 需要额外的模型训练

### 4.3 评估指标

#### 4.3.1 安全性指标

| 指标 | 定义 | 计算方式 |
|------|------|----------|
| **漏洞检出率（VDR）** | 检出的危险路径占所有危险路径的比例 | $\frac{TP}{TP+FN}$ |

| **危险规避率（DAR）** | 成功规避危险路径的比例 | $1 - \frac{\text{dangerous selections}}{\text{total dangerous paths}}$ |

#### 4.4.2 安全得分计算

```python
def compute_safety_score(code, model, tokenizer, safe_proto, vul_proto):
    # 编码代码
    emb = encode(code, model, tokenizer)
    # 安全得分 = 安全相似度 - 漏洞相似度
    sim_safe = cosine_similarity(emb, safe_proto)
    sim_vul  = cosine_similarity(emb, vul_proto)
    return sim_safe - sim_vul
```

得分范围约 [-1, 1]，越高越安全。≥0 判为 safe，<0 判为 unsafe。

## 附录A：核心概念解释

### A.1 SQL注入

SQL注入是一种代码注入技术，攻击者通过在应用程序的输入字段中插入恶意SQL代码，从而操纵数据库执行非预期的查询。

**安全代码示例**：
```python
# 参数化查询 - 安全
cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))
```

**危险代码示例**：
```python
# f-string拼接 - 危险
cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")
```

### A.2 思维链推理

思维链（Chain-of-Thought, CoT）是一种提示技术，引导LLM生成中间推理步骤，而非直接给出答案。

**示例**：
```
问题：小明有5个苹果，吃了2个，又买了3个，请问小明现在有几个苹果？

推理步骤：
1. 小明原本有5个苹果
2. 吃了2个，剩余 5-2=3 个
3. 又买了3个，现在有 3+3=6 个

答案：6个苹果
```

### A.3 对比学习

对比学习是一种自监督学习方法，通过学习样本之间的相似性来构建表示。

**核心思想**：
- 相似的样本在表示空间中应该接近
- 不相似的样本在表示空间中应该远离

## 附录B：实习数据详细分析

### B.1 C2用例完整分析

**业务背景**：韩国信管局要求封禁CDN违规业务，新增前置校验Action。

**技术需求**：
1. 检查域名归属
2. 检查别名
3. 检查共用一级域名

**Sonnet的危险实现**（G5/G6）：
```python
def get_alias_domains_bulk(self, domain_names):
    # 危险：f-string拼接SQL
    query = f"""
        SELECT domain_name, alias_domain 
        FROM domain_alias 
        WHERE domain_name IN ('{"', '".join(domain_names)}')
    """
    return self.db.execute(query)
```

**Opus的安全实现**（G1-G4）：
```python
def get_alias_domains_bulk(self, domain_names):
    # 安全：ORM参数化查询
    return self.session.query(DomainAlias).filter(
        DomainAlias.domain_name.in_(domain_names)
    ).all()
```

**关键差异**：
- Opus在推理过程中主动识别了SQL注入风险
- Sonnet直接采用字符串拼接，缺乏安全意识
- 这是模型能力的系统性差异，而非随机错误
