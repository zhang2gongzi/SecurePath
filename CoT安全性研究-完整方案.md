### 0.2 核心叙事框架

#### 第一步：发现问题（你的独特贡献）

> 在AI编程助手的代码生成任务中，我们发现一个系统性现象：LLM在生成代码时，可能输出多条不同的推理路径，其中某些路径生成的代码虽然功能正确，但存在安全漏洞（如SQL注入、缓冲区溢出等）。现有CoT路径选择方法（如Self-Consistency）无法识别这种安全风险。

#### 第二步：定义问题

> 现有CoT路径选择方法（如Self-Consistency）只关注答案频率或语义相似度，无法识别代码安全性。在代码生成场景中，我们需要一种新的机制：从多条候选代码中选择最安全的版本。

#### 第三步：解决问题

> 本文提出SecurePath框架，采用三阶段技术路线：
> 1. **阶段一**：对比学习预训练安全感知编码器（CodeBERT + InfoNCE）
> 2. **阶段二**：冻结编码器 + MLP训练安全分类器（768→256→128→2）
> 3. **阶段三**：分类器驱动的安全路径选择（从多候选中选择P(vul)最低的代码）

#### 第四步：验证效果

> 采用双实验设计互为印证：
> - 实验A（大规模自动评估）：在MSR测试集上验证分类器的代码安全性判别力（原型基线作为ablation）
> - 实验B（端到端验证）：在LLM实际代码生成场景中验证方法可用性

### 0.3 论文叙事框架（5章）

#### Chapter 1: 发现问题 — "LLM 生成的代码可能功能正确但存在漏洞"

**切入**：从实习中的真实案例出发——同一个"查询用户数据"需求，Sonnet 生成了 f-string SQL 拼接（可 SQL 注入），Opus 生成了参数化查询（安全）。这不是随机 bug，而是模型的**系统性差异**。

**引出现有方法的盲区**：Self-Consistency（多数投票）只看答案频率，不考虑安全性。一段代码被 8/10 条路径生成，不代表它安全——可能 8 条都忽视了同一个漏洞。

**核心故事钩子**：CoT 路径选择需要"安全意识"，现有方法全都没有。

#### Chapter 2: 相关工作 — "漏洞检测和路径选择是两条平行线"

| 领域 | 做什么 | 局限 |
|------|--------|------|
| 代码漏洞检测 | 静态分析、深度学习、CodeBERT fine-tune | 只管"有没有漏洞"，不参与路径选择 |
| CoT 路径选择 | Self-Consistency、CISC、Soft-SC | 只管"哪个答案最可能对"，不考虑安全 |

**Gap**：没有人把漏洞检测能力引入路径选择。本文填补这个空白。

#### Chapter 3: 方法 — "先学会看代码安不安全，再用来选路径"

**阶段一：安全感知编码器预训练（对比学习）**
- 在 MSR 数据集上使用 CodeBERT + 对比学习 + InfoNCE 损失
- 关键挑战：vul=0 记录中 func_before == func_after（92.7%），导致训练崩溃（准确率仅 52%）
- 解决方法（方案二）：从不同函数构建漏洞/安全代码池，确保语义差异
- 结果：pair matching 准确率 52% → 85.7%

**阶段二：安全分类器训练（冻结编码器 + AttentionPooling + MLP）**
- 关键发现：直接使用原型余弦相似度评分仅达 25%（远低于随机），安全/漏洞原型的余弦相似度 > 0.84
- 原因：对比学习优化的是配对区分，不优化全局 safe/vul 分类边界
- 方案：冻结编码器，提取全 token hidden states，训练 AttentionPooling + MLP（768 → 256 → 128 → 2）
- AttentionPooling 让模型学会关注安全关键 token（如内存操作、边界检查等），而非简单取 CLS
- 原型方法保留作为实验 ablation baseline，证明"简单相似度不足以判别代码安全性"

**阶段三：分类器驱动的安全路径选择**
- 用户提出代码生成请求 → LLM 生成 K 条候选代码
- 对每条候选：编码器提取全token hidden states → AttentionPooling → 分类器输出 P(vul)
- 选择安全概率最高的代码输出

#### Chapter 4: 实验 — "双实验设计，互为印证"

| | 实验 A：大规模自动评估 | 实验 B：端到端验证 |
|------|------|------|
| 数据 | MSR 测试集（43k 已标注） | LLM 生成（15 prompt × 5 候选 = 75 条） |
| 标注 | 无需人工，已有客观标签 | Bandit 预筛 + 少量人工确认 |
| 指标 | Acc / F1 / AUC | 安全选择率 vs Random / Self-Consistency |
| 方法 | 原型基线 + 安全分类器双方法对比 | 分类器安全概率排序 |
| 证明什么 | ① 原型不足(51%) ② 分类器有效 | 实际可用性 |
| 设计理由 | 数据量足够大、标签客观、不受 LLM 质量干扰 | 证明方法在真实场景中有效 |

**实验A的关键ablation**：原型余弦评分仅 51% Acc → 证明"简单 embedding 相似度不够" → 分类器学习非线性决策边界。原型失败不是废数据，而是论文中最有力的 ablation 证据。

**两个实验的逻辑关系**：A 证明"方法能区分"，B 证明"方法能用"。单一实验各有局限：
- 只用 A：审稿人会说"你在测试集上跑得好，实际代码生成中呢？"
- 只用 B：样本量小，且 LLM 生成质量是混杂变量
- A + B：判別力有大规模客观数据支撑，可用性有端到端场景验证

#### Chapter 5: 讨论与结论
- 方法适用范围：当前聚焦 C/C++ 内存安全问题，扩展到脚本语言需重新训练
- 与现有工具的关系：静态分析（规则） vs 嵌入空间判别（模式），可互补
- 未来方向：多语言扩展、更多漏洞类型

## 一、研究背景与动机

### 1.1 应用场景：代码生成任务的安全问题

**应用场景定义**：本文聚焦于LLM代码生成任务中的安全问题。

当用户向LLM提出代码生成请求时：
```
用户："帮我写一个根据用户ID查询用户数据的函数"

LLM可能生成多条候选代码：

路径A输出：
def get_user(user_id):
    query = f"SELECT * FROM users WHERE id={user_id}"  # SQL注入漏洞！
    return db.execute(query)

路径B输出：
def get_user(user_id):
    query = "SELECT * FROM users WHERE id=?"
    return db.execute(query, (user_id,))  # 安全：参数化查询
```

**核心问题**：现有CoT路径选择方法无法识别路径A存在SQL注入漏洞，可能选择危险代码。

### 1.2 MSR_data_cleaned数据集分析

数据集揭示的现象：
- 总函数数：188,636
- 漏洞函数：10,900 (5.8%)
- 非漏洞函数：177,736 (94.2%)

**6种内存安全漏洞分布**：
| CWE | 漏洞类型 | 数量 | 占比 |
|-----|----------|------|------|
| CWE-119 | 缓冲区溢出 | 26,497 | 48.7% |
| CWE-416 | Use After Free | 9,780 | 18.0% |
| CWE-125 | 越界读取 | 7,717 | 14.2% |
| CWE-476 | 空指针解引用 | 4,572 | 8.4% |
| CWE-190 | 整数溢出 | 3,671 | 6.7% |
| CWE-787 | 越界写入 | 2,918 | 5.4% |

### 1.3 研究机会

通过对比学习，可以让模型学习区分漏洞代码与安全代码的表示：

- **训练数据**：漏洞代码(func_before) vs 修复代码(func_after)
- **学习方法**：对比学习，让漏洞代码聚集，安全代码聚集，两者分离
- **应用**：评估LLM生成的多条候选代码，选择最安全的版本

**核心假设**：漏洞代码和安全代码在向量空间中具有不同的分布特征，通过对比学习可以有效区分。

### 3.2 阶段一：安全感知编码器训练

#### 3.2.1 数据准备

**数据来源**：MSR_data_cleaned（188,636函数）

**采样策略**：每种CWE类型采样1000条，共6000条样本

**标签构建**：
- 漏洞代码(func_before) → label=1 (危险)
- 修复代码(func_after) → label=0 (安全)

**数据划分**：
- 策略一（RQ3）：按比例划分（训练70%/验证10%/测试20%）
- 策略二（RQ4）：留一法（5种CWE训练，1种CWE测试）

#### 3.2.2 对比学习框架

**编码器**：CodeBERT-base

**投影头**：768维 → 256维

**对比损失**：InfoNCE Loss

```
正样本对：同类漏洞代码聚集
负样本对：漏洞代码 vs 修复代码分离
```

**训练目标**：
- 漏洞代码与安全代码在向量空间中形成可区分的表示
- 为后续分类器提供有判别力的 token-level 表示

#### 3.2.3 原型构建（作为实验 baseline）

**漏洞原型**：各CWE类型漏洞代码向量的均值
$$p_{vul}^{(cwe)} = \frac{1}{N}\sum_{i=1}^{N} Encoder(code_{vul}^{(i)})$$

**安全原型**：各CWE类型修复代码向量的均值
$$p_{safe}^{(cwe)} = \frac{1}{N}\sum_{i=1}^{N} Encoder(code_{safe}^{(i)})$$

**论文角色**：原型余弦相似度评分作为实验A的 baseline，验证"简单的 embedding 距离不足以判别代码安全性"。实验发现原型间余弦相似度 > 0.95，分类准确率仅 51%（接近随机），为后续分类器训练提供了动机。

### 3.3 阶段二：安全分类器训练

#### 3.3.1 动机

原型评分在 MSR 测试集上仅达 51% 准确率，证明安全代码判别不是简单的"在嵌入空间中靠近哪个原型"的问题。需要学习非线性的分类决策边界。

#### 3.3.2 分类器架构

在冻结的编码器之上训练 AttentionPooling + MLP：

```
CodeBERT (frozen) → 全token hidden states (512 × 768)
     → AttentionPooling (可训练): 每个token打分 → softmax → 加权求和
     → Linear(768, 256) → ReLU → Dropout(0.3)
     → Linear(256, 128) → ReLU → Dropout(0.3)
     → Linear(128, 2) → Softmax → P(vul | code)
```

**设计选择**：
- 使用可训练 AttentionPooling 替代固定的 CLS pooling，让模型学会关注安全相关 token（如 `memcpy`、`malloc`、缓冲区大小变量等）
- AttentionPooling 参数量 ~590K（768×768 + 768×1），轻量但提供显著表达能力
- 冻结编码器，仅训练 AttentionPooling + MLP（总参数量 ~820K），训练快速且防止过拟合
- 注意力权重可解释：可视化哪些 token 被模型关注来判别安全性

### 3.4 阶段三：分类器驱动的安全路径选择

#### 3.4.1 应用流程

```
1. 用户提出代码生成请求
   例："帮我写一个查询用户数据的函数"

2. LLM采样生成K条推理路径
   路径1 → 代码1
   路径2 → 代码2
   ...
   路径K → 代码K

3. 分类器评估每段代码的安全性
   P(vul | code_i) = Classifier(Encoder(code_i))

4. 选择安全概率最高的代码输出
   code* = argmin P(vul | code_i)
```

#### 3.4.2 安全得分计算

$$SafetyScore(code) = 1 - P(vul | code) = P(safe | code)$$

分数越高表示代码越安全。分类器输出的概率天然在 [0, 1]，比原始余弦相似度更具可解释性。

### 4.2 基线方法

#### 4.2.1 路径选择基线

| 方法 | 描述 | 特点 |
|------|------|------|
| Self-Consistency | 多数投票选择答案频率最高的 | 不考虑安全性 |
| Random Selection | 随机选择一条路径 | 基线对照 |
| Highest Confidence | 选择模型置信度最高的 | 传统方法 |
| CISC | 置信度加权自一致性（ACL 2025） | 置信度加权 |
| Soft-SC | 连续似然分数替代离散投票（ACL 2024） | 连续评分 |
| **SecureProtoCoT（本文）** | 安全感知的分类器路径选择 | 本文方法 |

#### 4.2.2 安全检测基线

| 方法 | 描述 | 特点 |
|------|------|------|
| Rule-based Detection | 基于规则的安全检测 | 误报高 |
| Static Analysis | 静态代码分析（Bandit等） | 仅代码层面 |
| LLM-as-Judge | 用LLM判断安全性 | 成本高 |
| **SecureProtoCoT（本文）** | 嵌入空间安全检测 | 高效、可解释 |

### 4.3 评估指标

#### 4.3.1 安全性指标

| 指标 | 定义 | 计算方式 |
|------|------|----------|
| **漏洞检出率（VDR）** | 检出的危险路径占所有危险路径的比例 | $\frac{TP}{TP+FN}$ |
| **误报率（FPR）** | 错误标记为危险的路径比例 | $\frac{FP}{FP+TN}$ |
| **安全选择准确率（SSA）** | 最终选择的路径为安全的比例 | $\frac{\text{safe selections}}{\text{total selections}}$ |
| **危险规避率（DAR）** | 成功规避危险路径的比例 | $1 - \frac{\text{dangerous selections}}{\text{total dangerous paths}}$ |

#### 4.3.2 功能性指标

| 指标 | 定义 | 计算方式 |
|------|------|----------|
| **任务完成率（TCR）** | 选择的路径能完成任务的比例 | 人工评估 |
| **推理质量（RQ）** | 推理逻辑的连贯性和正确性 | 专家评分1-5 |

#### 4.3.3 效率指标

| 指标 | 定义 | 计算方式 |
|------|------|----------|
| **推理延迟** | 从输入到输出的时间 | 毫秒 |
| **计算开销** | 额外的编码器前向传播次数 | 次数 |
| **API成本** | 相比基线方法的成本变化 | 美元 |

### 4.4 实验设计：双实验结构

#### 设计理念

单一实验无法同时保证"可信度"和"实用性"：

| 如果只用… | 审稿人会说… |
|-----------|------------|
| MSR 测试集自动评估 | "测试集上跑得好，实际代码生成中呢？" |
| LLM 生成代码人工评估 | "15 个任务 75 条样本，统计上可信吗？LLM 好坏是不是混杂了？" |
| **两者结合** | 大规模 + 客观标签证明判别力；端到端场景证明可用性 |

#### 实验 A：MSR 测试集自动评估

**目标**：在大规模、有客观标签的数据上验证安全分类器的代码安全性判别力（原型基线作为ablation）。

| 维度 | 内容 |
|------|------|
| 数据 | test_contrastive.csv（1,102条已标注漏洞/安全代码，6种CWE） |
| 方法 | 原型余弦评分（ablation）+ 冻结编码器MLP分类器（本文方法） |
| 指标 | Accuracy、Precision、Recall、F1、AUC |
| 成功标准 | 分类器 F1 > 0.75, AUC > 0.85；原型显著低于分类器 |
| 优势 | 数据量大、标签客观、无需 LLM、无需人工标注 |

#### 实验 B：LLM 生成代码端到端验证

**目标**：证明方法在 LLM 实际代码生成场景中可用。

| 维度 | 内容 |
|------|------|

| 指标 | 安全选择率 |

#### 4.4.2 安全得分计算

```python
def compute_safety_score(code, encoder, classifier, tokenizer):
    # 冻结编码器提取全token hidden states
    enc = tokenizer(code, max_length=512, padding=True, truncation=True, return_tensors='pt')
    hidden_states = encoder(input_ids=enc['input_ids'], 
                           attention_mask=enc['attention_mask']).last_hidden_state
    # AttentionPooling + MLP 输出 P(vul | code)
    logits = classifier(hidden_states, enc['attention_mask'])
    p_vul = softmax(logits)[:, 1]
    return 1.0 - p_vul  # 安全得分 = P(safe | code)
```

安全得分 ∈ [0, 1]，越高越安全。选择安全得分最高的候选代码输出。

#### 实验 A-PerCWE：按CWE类型分组评估

**目标**：分析分类器在6种CWE类型上的性能差异，识别哪些漏洞类型更容易/更难判别。

| 维度 | 内容 |
|------|------|
| 数据 | test_contrastive.csv（含cwe_type字段） |
| 方法 | 加载已训练分类器 → 对每条测试样本预测 → 按CWE分组计算Acc/Prec/Recall/F1/AUC |
| 指标 | 每种CWE的F1、AUC，总体加权平均 |
| 脚本 | `eval_per_cwe.py` |

#### 实验 A-LeaveOneOut：留一法跨CWE泛化

**目标**：证明分类器对"未见过的漏洞类型"的泛化能力。对每种CWE，用其余5种训练，在留出的1种上测试。

| 维度 | 内容 |
|------|------|
| 数据 | 全部train+val+test合并，按CWE重新划分 |
| 方法 | 对每种CWE：用5种CWE训练MLP分类器 → 在留出CWE上评估 |
| 指标 | 每种CWE的F1/AUC，6次实验的平均值 |
| 成功标准 | 留一法平均F1与全量训练F1差距 < 0.10 |
| 脚本 | `eval_leave_one_out.py` |
| 训练次数 | 6次（对应6种CWE） |

**关键发现**：vul=0 记录中 92.7% 的 func_before==func_after，若直接配对使用会导致标签矛盾（同一段代码同时出现为正样本和负样本），训练准确率仅 52%。

**解法（方案二）**：从不同函数构建漏洞/安全代码池，确保每段代码只出现一次，保证语义差异 → 对比学习准确率提升至 85.7%。

#### 12.1.2 方法对比

**原型基线（Cosine Similarity Scoring）**：
$$SafetyScore(code) = \cos(emb, p_{safe}) - \cos(emb, p_{vul})$$

**本文方法（Frozen Encoder + MLP Classifier）**：
$$P(vul \mid code) = \text{MLP}(\text{CodeBERT}_{frozen}(code))$$

#### 12.1.4 关键发现

1. **原型评分系统性失败**：AUC=0.0966（远低于 0.50），Recall/Presision 接近 0。原因是原型方法只考虑"靠近哪个均值"，无法捕获 safe/vul 代码在嵌入空间中的非线性决策边界。
2. **分类器显著有效**：F1=0.8840, AUC=0.9488，证明在编码器之上学习非线性分类边界是必要的。
3. **高 Recall (0.9056)**：漏报率低，适合安全场景（宁可误判不可漏判）。
4. **数据平衡 1:1 是关键**：之前不平衡训练（2100 safe vs 1275 vul）时 Precision 仅 0.64，平衡后提升至 0.86。

#### 12.1.5 Ablation 意义

原型→分类器的对比构成了论文最有力的 ablation：
- 原型是对比学习和原型学习的直接应用（SimCSE + Prototypical Networks），方法直观
- 原型完全失效（Acc 25%）证明"简单嵌入距离不足以判别安全性"
- 分类器的显著提升证明了"两阶段训练（预训练+微调）"的必要性
- 这一发现本身就是贡献：揭示了代码安全判别不能简化为嵌入空间中的邻近计算

## 附录A：核心概念解释

### A.1 SQL注入

SQL注入是一种代码注入技术，攻击者通过在应用程序的输入字段中插入恶意SQL代码，从而操纵数据库执行非预期的查询。

**安全代码示例**：
```python
# 参数化查询 - 安全
cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))
```

**危险代码示例**：
```python
# f-string拼接 - 危险
cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")
```

### A.2 思维链推理

思维链（Chain-of-Thought, CoT）是一种提示技术，引导LLM生成中间推理步骤，而非直接给出答案。

**示例**：
```
问题：小明有5个苹果，吃了2个，又买了3个，请问小明现在有几个苹果？

推理步骤：
1. 小明原本有5个苹果
2. 吃了2个，剩余 5-2=3 个
3. 又买了3个，现在有 3+3=6 个

答案：6个苹果
```

### A.3 对比学习

对比学习是一种自监督学习方法，通过学习样本之间的相似性来构建表示。

**核心思想**：
- 相似的样本在表示空间中应该接近
- 不相似的样本在表示空间中应该远离

### B.1 C2用例完整分析

**业务背景**：韩国信管局要求封禁CDN违规业务，新增前置校验Action。

**技术需求**：
1. 检查域名归属
2. 检查别名
3. 检查共用一级域名

**Sonnet的危险实现**（G5/G6）：
```python
def get_alias_domains_bulk(self, domain_names):
    # 危险：f-string拼接SQL
    query = f"""
        SELECT domain_name, alias_domain 
        FROM domain_alias 
        WHERE domain_name IN ('{"', '".join(domain_names)}')
    """
    return self.db.execute(query)
```

**Opus的安全实现**（G1-G4）：
```python
def get_alias_domains_bulk(self, domain_names):
    # 安全：ORM参数化查询
    return self.session.query(DomainAlias).filter(
        DomainAlias.domain_name.in_(domain_names)
    ).all()
```

**关键差异**：
- Opus在推理过程中主动识别了SQL注入风险
- Sonnet直接采用字符串拼接，缺乏安全意识
- 这是模型能力的系统性差异，而非随机错误
