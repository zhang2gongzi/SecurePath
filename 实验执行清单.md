**方案二（实际采用）**：

| 数据池 | 来源 | 数量 |

#### 4.2 RQ3：CoT路径安全选择实验（下一步）

##### 4.2.1 实验目标

验证编码器+原型能否在实际代码生成场景中有效识别并选择安全代码。

##### 4.2.2 实验设计思路

**为什么不能用 MSR 测试集直接评估？**

MSR 数据集的漏洞是 C/C++ 内存安全漏洞（缓冲区溢出、UAF等），与 LLM 代码生成任务中的常见漏洞（SQL注入、命令注入等）类型不同。因此需要 LLM 实际生成候选代码来做端到端验证。

**但可以结合两者**：MSR 测试集做定量评估（数据量大、标签客观），LLM 生成任务做定性验证（证明实际可用）。

```
评估 = 自动评估（MSR测试集，大规模） + 人工评估（LLM生成，端到端）
```

##### 4.2.3 实验A：MSR测试集自动评估（无人工标注）

直接在已有的 `test_contrastive.csv` 上评估原型判别能力：

| 步骤 | 内容 |
|------|------|
| 数据 | test_contrastive.csv（数千条已标注漏洞/安全代码） |
| 方法 | 对每条代码计算 SafetyScore = cosine(emb, safe_proto) - cosine(emb, vul_proto) |
| 指标 | Accuracy、Precision、Recall、F1、AUC（阈值=0） |
| 优势 | 数据量大，标签客观，无需人工标注 |

这能回答：**原型方法在已知漏洞类型上的判别能力如何？**

##### 4.2.4 实验B：LLM生成代码端到端评估（少量人工标注）

| 步骤 | 内容 |
|------|------|
| 测试任务 | 15-20 个代码生成 prompt |
| LLM采样 | 每任务 K=5 条候选（温度=0.8），共 75-100 条 |
| 自动预筛选 | Bandit/Semgrep 扫描，自动标记明显漏洞 |
| 人工标注 | 仅确认边界情况（预计 30-40 条需人工） |
| 评估 | 安全选择率 vs Random baseline |

## 五、评估指标

### 5.1 漏洞检测指标
| 指标 | 定义 | 计算方式 |
|------|------|----------|
| Accuracy | 分类准确率 | (TP+TN)/(TP+TN+FP+FN) |
| Precision | 精确率 | TP/(TP+FP) |
| Recall | 召回率 | TP/(TP+FN) |
| F1 | F1分数 | 2*P*R/(P+R) |
| AUC | ROC曲线下面积 | sklearn计算 |

### 5.2 表示学习指标
| 指标 | 定义 | 计算方式 |
|------|------|----------|
| Silhouette Score | 聚类质量 | sklearn.metrics.silhouette_score |
| Davies-Bouldin | 聚类紧密度 | sklearn.metrics.davies_bouldin_score |

### 5.3 CoT路径选择指标
| 指标 | 定义 | 计算方式 |
|------|------|----------|
| 安全选择率 | 选择安全路径的比例 | 安全选择数/总选择数 |
| 漏洞规避率 | 成功规避漏洞的比例 | 1 - 危险选择数/总危险路径 |
